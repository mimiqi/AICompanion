# 2026-04-19 20:02 - 收尾上轮遗留的后台 OLV 进程

## 一、本次会话目标

接续 18:33 的"通链验证准备"会话。上轮我以"后台 + 不主动结束"方式启动了 OLV 主服务用于验证,但会话结束时**没有显式停止它**,导致用户在自己的终端 1 里于 20:00:45 重新执行 `uv run run_server.py` 时撞到端口冲突 `Errno 10048: ('::1', 12393)`。本次的目标是回收上轮遗留进程、释放端口、确认状态干净,让用户能正常重新启动服务。

会话起点：用户终端 1 启动失败,我手中还持有上轮的后台 shell 句柄;`Get-Process` 看不到 python(权限受限),但 `Test-NetConnection` 显示 IPv4/IPv6 端口 12393 都还在监听。

会话终点：上轮 PID 33188 已被强制终止;`Get-NetTCPConnection` 确认 12393 在两个协议栈上都已释放;用户可在自己的终端无冲突重启 OLV。

## 三、关键诊断要点

### 3.1 为什么 `Get-Process python` 看不到?

非管理员的 PowerShell 会话用 `Get-Process` 时,无法跨用户/会话看到完整进程列表;但 `Get-NetTCPConnection -State Listen` 即使非 admin 也能拿到 `OwningProcess` 字段。**端口占用排查推荐先用 `Get-NetTCPConnection`,再去针对性 `Stop-Process -Force`,而不是按进程名扫描**。

### 3.2 为什么用户的进程能"启动成功"再失败?

uvicorn 的双协议栈绑定是**先后顺序**:
1. 先 bind `127.0.0.1:12393` — 此时我的旧服务在 IPv4 已经退到 TIME_WAIT 或者刚释放,**用户绑定成功** → 打印 `Application startup complete`
2. 再 bind `::1:12393` — 我的旧服务还活着,IPv6 一直占着 → bind 失败 → uvicorn 整体回滚 → `Application shutdown complete`

所以日志里能看到 `Application startup complete` 紧跟着 `Errno 10048`,看起来像"启动了又挂",实际是"半启动失败回滚"。

### 3.3 我的设计失误

上轮我用了 `block_until_ms: 0`(立即后台化)启动 OLV 用作 HTTP/health 探测,验证完后**应该主动 `Stop-Process` 收尾**,但我没做。这是工具调用习惯问题,后续启动任何长时常驻服务前,都要先在心里规划"何时收回"。

## 六、对未来 AI 会话的约定 (自我提醒)

- 任何用 `block_until_ms: 0` 后台启动的常驻服务(uvicorn / WS server / daemon),**会话结束前必须主动 Stop-Process 或 taskkill 回收**
- 端口排查首选 `Get-NetTCPConnection -LocalPort N -State Listen` 而非 `Get-Process`
- 双协议栈端口冲突的特征是 `Application startup complete` 紧跟 `Errno 10048`,排查方向直接锁定 IPv6
