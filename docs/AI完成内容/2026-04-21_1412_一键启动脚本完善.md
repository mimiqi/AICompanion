## 五、关键技术决策

### 为什么用轮询 `Wait-Port` 而不是固定 `Start-Sleep`

旧脚本 `Start-Sleep -Seconds 10` 在两种场景下都不对:
- SoVITS 在你的 Windows + RTX 显卡上其实 4-6 秒就起来,等 10 秒浪费 4 秒
- OLV 首次启动时 faster-whisper 要下载 large-v3-turbo 模型(~1.5GB),固定等 8 秒铁定不够,导致后续操作(开浏览器)在 OLV 还没就绪时执行

新的 `Wait-Port -Port N -TimeoutSec M -Name 'XXX'`:
- 每秒 `Get-NetTCPConnection -LocalPort N -State Listen` 检查一次
- 一旦端口出现 LISTEN 立即返回 `$true` 并打印实际花费秒数
- 超时打 warning 返回 `$false`,主流程根据结果决定是否继续

### 为什么 `-ForceKill` 默认 OFF

端口被占可能是**别的服务在用**(比如另一个 VS Code 调试 server 占了 12393),无脑 kill 会打断用户其他工作。默认报错 + 提示用户 `\stop-all.ps1` 或 `-ForceKill`,把决策权留给用户。

### 为什么 OLV 窗口要 inject uv PATH

我们之前撞过几次:OLV 窗口启动后 `uv` 命令找不到,因为新窗口的 PATH 不一定包含 `C:\Users\mimiqi\.local\bin`(uv 安装位置)。每次让用户手动 `$env:Path = ...` 太烦。新脚本在 OLV 启动命令里硬编码:

```powershell
$env:Path = 'C:\Users\mimiqi\.local\bin;' + $env:Path ; uv run run_server.py
```

### 为什么 `Start-In-NewWindow` 把多行命令拼成单行

PowerShell 的 here-string `@" ... "@` 在 `Start-Process -ArgumentList` 透传时容易被空格/换行破坏,导致新窗口收到的命令不完整。统一拼成单行 + `;` 分隔最稳:

```powershell
$oneLine = ($Command -split "`r?`n" | Where-Object { $_.Trim() -ne "" }) -join " ; "
```
