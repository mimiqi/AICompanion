## 三、设计要点

### 3.1 关键 invariant:Live2D 模式 100% 行为不变

- 现有 `mao_pro` 条目不加任何字段 → `setModelInfo` 中 `skinType ?? 'live2d'` → 走完全原有逻辑
- `App.tsx` 默认渲染 `<Live2D />`
- `use-audio-task.ts` 的 `if (skinType !== 'live2d')` 分支不会进入,原有 ~120 行 Live2D 处理代码原封不动

### 3.2 静态模式的"音频管线保活"陷阱

`Live2D` 组件内部调了 `useAudioTask()` / `useInterrupt()` / `useIpcHandlers()`(原 live2d.tsx 第 46 行)。**如果只是 `skinType !== 'live2d' ? null : <Live2D/>`,音频回放会彻底停止**。

解决:`StaticAvatar` 组件也必须调用同样的三个 hook。这样无论哪种皮肤,音频/中断/IPC 链路都保持活着。

```tsx
// static-avatar.tsx 顶部
useIpcHandlers();
useInterrupt();
useAudioTask();
```

### 3.3 audioManager 的 null model 兼容性

读 [audio-manager.ts](../../frontend_overlay/upstream/src/renderer/src/utils/audio-manager.ts) 后发现它对 `setCurrentAudio(audio, null)` 已经有 graceful 处理:

```20:50:frontend_overlay/upstream/src/renderer/src/utils/audio-manager.ts
  stopCurrentAudioAndLipSync() {
    if (this.currentAudio) {
      ...
      const model = this.currentModel;
      if (model && model._wavFileHandler) {
        // ...lip sync stop...
      } else if (model) {
        console.warn('[AudioManager] Current model does not have _wavFileHandler...');
      } else {
        console.log('[AudioManager] No associated model found to stop lip sync.');
      }
```

所以静态模式可以直接 `audioManager.setCurrentAudio(audio, null)`,interrupt 信号到达时也能正常停止音频(只是不调 lip sync)。**不需要改 audio-manager**。

### 3.4 sprite 索引匹配的 fallback 链

`StaticAvatar` 选图逻辑:

```ts
sprites[String(staticExpression)]   // 精确匹配 emotion index
  ?? sprites['0']                   // 默认 sprite (key '0')
  ?? staticUrl                      // 兜底单图
  ?? ''                             // 都没有 → 显示提示文字
```

这样即使 LLM 输出的 emo 标签对应的 sprite 没准备(如只准备了 5 张 sprite 但 emo 是 6),也能 fallback 不报错。

### 3.5 setModelInfo 的多模式校验

```ts
const validLive2d       = skinType === 'live2d'        && Boolean(info.url);
const validStaticSingle = skinType === 'static_single' && Boolean(info.staticUrl);
const validStaticMulti  = skinType === 'static_multi'
  && (Boolean(info.staticUrl) || (info.sprites && Object.keys(info.sprites).length > 0));

if (!validLive2d && !validStaticSingle && !validStaticMulti) {
  console.warn(`[Live2DConfig] Invalid model info for skinType=${skinType}, clearing.`);
  return;
}
```

每种 skinType 有独立的最小字段要求。错误配置会在 console 显式 warn,而不是静默崩溃。

## 六、未来如何启用利托里奥皮肤(给将来的你)

走 static_multi 方案的步骤:

1. 准备 8 张利托里奥表情立绘 PNG(透明背景,推荐 720×1280 或 1080×1920),命名为 `neutral.png` / `joy.png` / `smirk.png` / `anger.png` / `sadness.png` / `surprise.png` / `disgust.png` / `fear.png`
2. 把它们放到 `Open-LLM-VTuber/characters/littorio/`(目录如果被 OLV 的 .gitignore 排除了,需要类似处理 backgrounds/ 那样加例外)
3. 创建对应的静态资源路由(可能需要在 OLV server.py 加 `/characters` 静态挂载,或者复用现有 `/bg` 把图放 backgrounds 下)
4. 修改 `Open-LLM-VTuber/conf.yaml` 的 `live2d_model_name: 'mao_pro'` → `'littorio_static_multi'`
5. 重启 OLV → 桌宠变成利托里奥静态立绘,LLM 触发不同 emo 时图片切换

走 Live2D 方案的步骤(找到模型后):

1. 把 Live2D 资源放到 `Open-LLM-VTuber/live2d-models/littorio/runtime/`
2. 在 `model_dict.json` 加新条目,`skinType` 不写(默认 live2d),`url` 指向 `littorio.model3.json`
3. 修改 conf.yaml 切到新条目
4. 不需要任何代码改动,本次预留的 skinType 系统就为了这种场景设计
